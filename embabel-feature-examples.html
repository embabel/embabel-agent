Feature 1

Personalized Horoscope News Brief

A user gives a casual prompt such as "Lynda is a Scorpio" and receives a short, amusing Markdown writeup that connects a horoscope to current news stories.

The product outcome is not "run a prompt." The outcome is a personalized, timely brief from minimal user input.

When extractStarPerson does not add StarPerson to the blackboard, the goal is still unsatisfied.

This is where Embabel's GOAP loop matters. After every action, the process observes the current blackboard, rebuilds the world state, searches for a low-cost action sequence to a goal, executes the first action in that sequence, and repeats. extractStarPerson is not retried forever because it has run and the default @Action(canRerun=false) setting makes actions one-shot for a process. The next planning cycle therefore looks for another way to produce the still-missing StarPerson.

The agent class is the capability boundary. Embabel registers the actions in the class as one news-briefing capability.

@SecureAgentTool is the security boundary for remote tool use. It attaches a Spring Security expression to the exported capability, so a caller must have news:read authority before this agent can be used as a secured tool.

When tools need request metadata @SecureAgentTool decides whether the remote capability may be called. If a tool also needs infrastructure data such as a tenant ID, auth token, or correlation ID during execution, pass it through ToolCallContext. That keeps operational metadata out of the LLM-visible tool schema.

@Action
fun produceDigest(topic: NewsTopic, context: OperationContext): NewsDigest {
    val focusClause = if (topic.focusArea.isNotBlank())
        "Focus specifically on: ${topic.focusArea}."
    else ""

    val digestItemList = context.ai()
        .withDefaultLlm()
        .withToolGroup(CoreToolGroups.WEB)
        .createObject<DigestItemList>(
            """
            Use web search to find the 5 most recent and relevant news items about: ${topic.topic}.
            $focusClause
            For each item return a headline, a 2-3 sentence summary, and the source URL.
            """.trimIndent()
        )

    val narrative = context.ai()
        .withDefaultLlm()
        .createObject<String>(
            """
            Write a 3-sentence editorial summary of these news items about ${topic.topic}.
            Be concise and objective.

            Items:
            ${digestItemList.items.joinToString("\n") { "- ${it.headline}: ${it.summary}" }}
            """.trimIndent()
        )

    return NewsDigest(
        topic = topic.topic,
        items = digestItemList.items,
        content = narrative,
    )
}

A user asks a serious research question. The system compares independent model outputs, merges the best material, critiques the merged report, and retries if the result is not good enough.

The product outcome is a higher-confidence research answer: independent evidence gathering, one merged answer, an explicit review, and a retry path when the review rejects it.

4.7 Retry After Rejection

If the critic rejects the answer, REPORT_UNSATISFACTORY becomes true. The redo actions are then valid because of their pre condition, and they receive the Critique so the next research pass can address the reviewer's reasoning.

The loop is not an automatic wait-until-success retry. Each pass writes new facts to the blackboard, then the planner chooses the next valid action. Rejection makes redo research valid; acceptance makes the final goal action valid.

This retry loop is still a good fit for ordinary GOAP because the loop is driven by facts: report, critique, condition, retry. If the workflow becomes a set of named phases such as "draft", "review", "revise", and "await human approval", model those phases with @State so the planner only considers actions for the current phase.

Design Heuristics

Start from the user outcome. Then choose the smallest Embabel concepts that make the workflow explicit, typed, inspectable, and testable.
  1. If the feature has a clear final user-visible result, mark that action with @AchievesGoal.
  2. If the feature has meaningful intermediate artifacts, model them as domain types and connect them with action signatures.
  3. If the LLM needs current external data, add tool groups only to the action that needs them.
  4. If the workflow depends on runtime quality or approval, use conditions rather than hiding routing in one method.
  5. If multiple same-type objects coexist, use named bindings.
  6. If the feature is repeated chat, use trigger actions and chatbot utility flow instead of forcing a terminal goal.
  7. If answers must be grounded in documents, use SearchOperations and ToolishRag.
  8. If something is deterministic, keep it in Kotlin or Java.
  9. If prompt behavior matters, test it with a fake operation context.

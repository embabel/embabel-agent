Feature 1

Personalized Horoscope News Brief

A user gives a casual prompt such as "Lynda is a Scorpio" and receives a short, amusing Markdown writeup that connects a horoscope to current news stories.

The product outcome is not "run a prompt." The outcome is a personalized, timely brief from minimal user input.

Source

  • StarNewsFinder.kt

Why Embabel Fits

The feature has a real workflow: understand user input, get domain data, search current news, and synthesize a final answer. Embabel expresses each product step as a typed action and lets the planner connect them through domain objects.

When extractStarPerson does not add StarPerson to the blackboard, the goal is still unsatisfied.

This is where Embabel's GOAP loop matters. After every action, the process observes the current blackboard, rebuilds the world state, searches for a low-cost action sequence to a goal, executes the first action in that sequence, and repeats. extractStarPerson is not retried forever because it has run and, by default, canRerun is false. The next planning cycle therefore looks for another way to produce the still-missing StarPerson.

  • Direct route: UserInput -> StarPerson
  • Fallback route: UserInput -> Person, then Person -> Starry, then Person + Starry -> StarPerson

// Product capability: remotely usable news digest.

@Agent(description = "Research a topic and return a curated news digest with headlines and summaries")

	            User input: ${userInput.content}
	            """.trimIndent()
	        )
    data class NewsTopic(
        val topic: String,
        val focusArea: String = "",
    )

}
// Product result: a completed digest visible to remote callers as "newsDigest".
// Embabel concept: Export exposes the goal, not every internal action.
@AchievesGoal(

    export = Export(
        remote = true,
        name = "newsDigest",
        startingInputTypes = [UserInput::class],

@Action
fun produceDigest(topic: NewsTopic, context: OperationContext): NewsDigest {
	    val digestItemList = context.ai()
	        .withDefaultLlm()
	        .withToolGroup(CoreToolGroups.WEB) // Only this call needs current news.
	        .createObject<DigestItemList>(
	            """
	            Use web search to find the 5 most recent and relevant news items about: ${topic.topic}.
	            $focusClause
	            For each item return a headline, a 2-3 sentence summary, and the source URL.
	            """.trimIndent()
	        )
    data class DigestItem(
        val headline: String,
        val summary: String,
        val url: String,
    )

    data class DigestItemList(
        val items: List<DigestItem>,
    )

	    val narrative = context.ai()
	        .withDefaultLlm()                  // No web tools needed for summarizing known items.

A business user asks for a market report on a company or sector and receives SWOT entries, competitor insights, key trends, and an executive summary.

The outcome is a decision-ready report assembled from current evidence.

3.1 Parse the Business Question into an Analysis Subject

The product starts from a human request such as "analyze the European EV charging market." The system first normalizes that into a typed subject and region so the rest of the workflow is stable.

// Product step: understand what market or company the user wants analyzed.
// Embabel concept: create a typed AnalysisSubject instead of carrying a raw prompt string.
@Action
fun parseSubject(
    userInput: UserInput,
    context: OperationContext,
): AnalysisSubject = context.ai()
    .withDefaultLlm()
    .createObject(
        """
        Extract the subject of market analysis from the following user input.
        Identify the company name, sector, or market segment, and the geographic region if mentioned.
        Default the region to Global if not specified.

        User input: ${userInput.content}
        """.trimIndent()
    )
data class AnalysisSubject(
    val subject: String,
    val region: String = "Global",
)

3.2 Gather Evidence Before Writing the Report

The report should be grounded in recent information. The web-enabled action creates a raw intelligence artifact as a stable evidence base.

// Product step: gather current evidence before analysis.
// Embabel concept: one web-enabled action creates rawIntelligence.
@Action
fun gatherIntelligence(
    subject: AnalysisSubject,
    context: OperationContext,
): String = context.ai()
    .withDefaultLlm()
    .withToolGroup(CoreToolGroups.WEB) // Current market info requires external search.
    .createObject(
        """
        Search the web to gather recent market intelligence about: ${subject.subject}
        Region: ${subject.region}

        Collect recent news, competitor moves, industry trends, and notable strengths or vulnerabilities.
        """.trimIndent()
    )

3.3 Synthesize Typed Sections from Shared Evidence

The final report is not one blob. It has sections with different jobs: SWOT, competitors, trends, and executive summary. Each LLM call receives the same evidence but has a narrower output contract.

Strengths Internal advantages the subject can build on, such as brand, technology, distribution, talent, or cost position.
Weaknesses Internal limitations that hold the subject back, such as high costs, capability gaps, poor execution, or fragile dependencies.
Opportunities External conditions the subject could exploit, such as market growth, regulation, new customer needs, or competitor weakness.
Threats External risks that could hurt the subject, such as new entrants, regulation, substitutes, supply constraints, or market decline. 
// Product result: a structured decision report.
// Embabel concept: the goal action consumes both the subject and gathered evidence.
@AchievesGoal(
    description = "Produce a structured market intelligence report with SWOT analysis, competitive insights, and key trends",
    export = Export(remote = true, name = "marketIntelligenceReport", startingInputTypes = [UserInput::class])
)
@Action
fun synthesiseReport(
    subject: AnalysisSubject,
    rawIntelligence: String,
    context: OperationContext,
): MarketIntelligenceReport {
    // Product section: SWOT entries.
    // SWOT = Strengths, Weaknesses, Opportunities, Threats.
    // It separates internal factors (strengths/weaknesses) from external factors (opportunities/threats).
    // Embabel concept: each call returns a typed object, not unstructured text.
    val swotList = context.ai().withDefaultLlm().createObject<SwotEntryList>(
        """
        Based on the following market intelligence, produce a SWOT analysis for ${subject.subject}.
        Each entry must have a category and concise description.
        Return 2-3 items per category.

        Intelligence:
        $rawIntelligence
        """.trimIndent()
    )
    data class SwotEntry(
        val category: String,      // Strength, Weakness, Opportunity, or Threat.
        val description: String,
    )

    data class SwotEntryList(val items: List<SwotEntry>)

    // Product section: competitor insights.
    val competitorList = context.ai().withDefaultLlm().createObject<CompetitorInsightList>(
        """
        Based on the following market intelligence, identify the top 4 competitors
        or comparable entities for ${subject.subject} in ${subject.region}.
        For each, provide a recent development or strategic positioning insight.

        Intelligence:
        $rawIntelligence
        """.trimIndent()
    )
    data class CompetitorInsight(
        val name: String,
        val insight: String,
    )

    data class CompetitorInsightList(val items: List<CompetitorInsight>)

    // Product section: market trends.
    val trendList = context.ai().withDefaultLlm().createObject<KeyTrendList>(
        """
        Based on the following market intelligence, extract the 5 most significant
        market trends affecting ${subject.subject} in ${subject.region}.
        Return each as a concise single sentence inside the items list.

        Intelligence:
        $rawIntelligence
        """.trimIndent()
    )
    data class KeyTrendList(val items: List<String>)

    // Product section: executive summary for decision makers.
    val summary = context.ai().withDefaultLlm().createObject<String>(
        """
        Write a 4-sentence executive summary for a market intelligence report on
        ${subject.subject} (${subject.region}).
        Be concise, analytical, and suitable for a senior decision-maker.

        Key findings:
        SWOT: ${swotList.items.joinToString("; ") { "${it.category}: ${it.description}" }}
        Trends: ${trendList.items.joinToString("; ")}
        """.trimIndent()
    )

    // Product assembly: normal code composes the final report from typed sections.
    return MarketIntelligenceReport(
        subject = subject.subject,
        region = subject.region,
        executiveSummary = summary,
        swot = swotList.items,
        competitors = competitorList.items,
        keyTrends = trendList.items,
        content = summary,
    )
}
data class MarketIntelligenceReport(
    val subject: String,
    val region: String,
    val executiveSummary: String,
    val swot: List<SwotEntry>,
    val competitors: List<CompetitorInsight>,
    val keyTrends: List<String>,
    override val content: String,
) : HasContent
Market Intelligence's key lesson: gather evidence once with tools, then use narrow typed synthesis calls. This makes the report easier to test, explain, and secure than a single all-purpose research prompt.

The product outcome is a higher-confidence report with an explicit quality gate.

Two model paths produce the same type, SingleLlmReport. The product wants both, not whichever one happens to be latest, so the blackboard entries are named.

4.3 Use Web and Browser Tools Inside the Shared Research Implementation

The model-specific actions delegate to one implementation. That keeps the model choice separate from the product behavior: use external sources and write a bounded report.

// Product behavior: answer precise questions with current sources.
// Embabel concept: PromptRunner combines model, contributors, and tools for this call.

4.4 Merge Named Reports

The merge action depends on the two named reports. @RequireNameMatch makes the code explicit: gpt4Report means the GPT report, not any SingleLlmReport on the blackboard.

// Product step: combine the strongest findings from both models.
// Embabel concept: @RequireNameMatch injects the specifically named reports.
@Action(
    post = [REPORT_SATISFACTORY],
    outputBinding = "mergedReport",
    canRerun = true,
)
fun mergeReports(
    userInput: UserInput,
    @RequireNameMatch gpt4Report: SingleLlmReport,
    @RequireNameMatch claudeReport: SingleLlmReport,
    context: OperationContext,
): ResearchReport {
    val reports = listOf(gpt4Report, claudeReport)
    return context.promptRunner(
        llm = LlmOptions(properties.criticModeName),
        promptContributors = properties.promptContributors,
    ).create(
        """
        Merge the following research reports into a single report taking the best of each.
        Consider the user direction: <${userInput.content}>

        ${reports.joinToString("\n\n") { "Report from ${it.model}\n${it.report.infoString(verbose = true)}" }}
        """.trimIndent()
    )
}

4.5 Turn Critique into Planning Facts

The product does not want to accept a bad report. The critique object is evaluated by conditions. Those conditions control whether the planner can finish or must redo research.

// Product quality gate: ask a critic model whether the report answers the request.
// Embabel concept: this creates Critique, which conditions evaluate.
@Action(post = [REPORT_SATISFACTORY], canRerun = true)
fun critiqueMergedReport(
    userInput: UserInput,
    @RequireNameMatch mergedReport: ResearchReport,
    context: OperationContext,
): Critique = context.ai().withLlm(properties.criticModeName)
    .create(
        """
        Is this research report satisfactory? Consider the following question:
        <${userInput.content}>
        The report is satisfactory if it answers the question with adequate references.

        ${mergedReport.infoString(verbose = true)}
        """.trimIndent(),
    )
data class Critique(
    val accepted: Boolean,
    val reasoning: String,
)

// Embabel concept: conditions convert the Critique value into named planning facts.
@Condition(name = REPORT_SATISFACTORY)
fun makesTheGrade(critique: Critique): Boolean = critique.accepted

@Condition(name = REPORT_UNSATISFACTORY)
fun rejected(critique: Critique): Boolean = !critique.accepted

4.6 Finish Only When the Quality Gate Passes

The final action has a precondition. If REPORT_SATISFACTORY is not true, this action is not valid.

// Product result: only expose the final report after the satisfactory condition is true.
// Embabel concept: pre = [REPORT_SATISFACTORY] gates goal completion.
@AchievesGoal(description = "Completes a research or question answering task, producing a research report")
@Action(pre = [REPORT_SATISFACTORY], outputBinding = "finalResearchReport")
fun acceptReport(
    @RequireNameMatch mergedReport: ResearchReport,
    critique: Critique,
) = mergedReport
Researcher's key lesson: conditions are not just booleans in code. They are planning facts. The planner can see "satisfactory" versus "unsatisfactory" and choose finish or retry paths accordingly.

5.2 Extract Assertions with Multiple Models, then Merge

The product risk is missing claims or duplicating claims. The aggregate stage asks multiple model configurations to extract assertions, then a merge step rationalizes the list.

flow {
    // Product step: identify factual claims in user content.
    // Embabel concept: aggregate runs multiple transforms from UserInput to FactualAssertions.
    aggregate<UserInput, FactualAssertions, RationalizedFactualAssertions>(
        transforms = llms.map { llm ->
            { context ->
                context.ai()
                    .withLlm(llm)
                    .withToolGroup(CoreToolGroups.WEB)
                    .createObject(
                        """
                        Given the following content, identify any factual assertions.
                        Phrase them as standalone assertions.
                        Do not duplicate assertions.

                        # Content
                        ${context.input.content}
                        """.trimIndent()
                    )
            }
        },
        data class FactualAssertion(
            val standaloneAssertion: String,
        )

        data class FactualAssertions(
            val factualAssertions: List<FactualAssertion>,
        )

        // Product step: turn overlapping model outputs into one clean list.
        // Embabel concept: merge receives all aggregate results and returns RationalizedFactualAssertions.
        merge = { list, context ->
            context.ai().withDefaultLlm()
                .createObject<RationalizedFactualAssertions>(
                    """
                    Given the following factual assertions, merge them into a single list if
                    any are the same. Condense into one assertion if one assertion negates another.

                    # Assertions
                    ${list.flatMap { it.factualAssertions }.joinToString("\n") { "- " + it.standaloneAssertion }}
                    """.trimIndent()
                )
        },
        data class RationalizedFactualAssertions(
            val factualAssertions: List<FactualAssertion>,
            val numberMerged: Int,
        )
    )
}

5.3 Check Assertions in Parallel with Web and Browser Tools

Once assertions are independent, they should not be checked serially. The action configures one prompt runner with the right tools and applies it to every assertion concurrently.

// Product step: check each assertion independently.
// Embabel concept: transformation maps RationalizedFactualAssertions to FactCheck.
transformation<RationalizedFactualAssertions, FactCheck> { operationContext ->
    val promptRunner = operationContext.ai()
        .withAutoLlm()
        // Product capability: assertions need current sources and sometimes browser inspection.
        // Embabel concept: tools are configured once and reused across the fan-out.
        .withTools(CoreToolGroups.WEB, CoreToolGroups.BROWSER_AUTOMATION)

    // Product performance: assertions do not depend on each other.
    // Embabel concept: parallelMap keeps each call inside the current operation context.
    val checks = operationContext.input.factualAssertions
        .parallelMap(operationContext) { assertion ->
            promptRunner.createObject<AssertionCheck>(
                """
                Given the following assertion, check if it is true or false and explain why.
                Express confidence as a number between 0 and 1.
                Use web tools.

Design Heuristics

Start from the user outcome. Then choose the smallest Embabel concepts that make the workflow explicit, typed, inspectable, and testable.
  1. If the feature has a clear final user-visible result, mark that action with @AchievesGoal.
  2. If the feature has meaningful intermediate artifacts, model them as domain types and connect them with action signatures.
  3. If the LLM needs current external data, add tool groups only to the action that needs them.
  4. If the workflow depends on runtime quality or approval, use conditions rather than hiding routing in one method.
  5. If multiple same-type objects coexist, use named bindings.
  6. If the feature is repeated chat, use trigger actions and chatbot utility flow instead of forcing a terminal goal.
  7. If answers must be grounded in documents, use SearchOperations and ToolishRag.
  8. If something is deterministic, keep it in Kotlin or Java.
  9. If prompt behavior matters, test it with a fake operation context.
